톡톡 로컬 watcher (주간) 작업입니다.

목표:
- 로컬 브라우저 기준으로 톡톡 목록 변화를 가볍게 감지합니다.
- 변화가 없으면 아무 것도 하지 않습니다.
- 변화가 있으면 현재 세션에 짧게 보고하고, 승인/후속 처리를 위한 로컬 상태 파일을 갱신합니다.

주간 스케줄:
- Asia/Seoul 기준 09:00~23:59
- 5분마다 실행

반드시 아래 순서대로 진행하세요.

[1] 로컬 설정 읽기
1. `./runtime-data/navertalk-local-config.json` 파일을 읽습니다.
2. 아래 값을 확보합니다.
   - `talkUrl`
   - `approvalChannelId`
   - `controlChannelId`
3. 값이 비어 있으면 현재 세션에 짧게 실패 사유를 보고하고 종료합니다.

[2] 전용 자동화 Chrome 준비
1. `./smartstore-cs-worker/scripts/start-automation-chrome.sh` 를 실행합니다.
2. 이어서 `./smartstore-cs-worker/scripts/check-automation-chrome.sh` 로 `RUNNING` 확인합니다.
3. 실행 실패면 현재 세션에 짧게 실패 사유를 보고하고 종료합니다.
4. 일반 Chrome(user 프로필)에는 붙지 않습니다. browser 도구를 쓰더라도 반드시 자동화 Chrome(9223)만 사용합니다.
5. 자동화 Chrome 창은 백그라운드 전용으로 취급합니다. 불필요하게 앞으로 가져오지 말고, 파싱/확인은 백그라운드 탭 기준으로 진행합니다.

[2-1] 로그인 해제 자동 복구
1. 로그인 확인을 위해 `https://sell.smartstore.naver.com/` 루트나 `#/comment/` 를 별도 새탭으로 열지 않습니다.
2. `talkUrl` 탭을 재사용하거나, 필요할 때만 백그라운드 임시 탭으로 `talkUrl` 을 확인합니다.
3. 그 결과가 `https://sell.smartstore.naver.com/#/home/about` 또는 `accounts.commerce.naver.com/login...` 이면 로그인 필요 상태로 봅니다.
4. 이 경우 `로그인하기` 버튼과 `김*호 님 / 네이버 아이디로 간편 로그인 / sos8*** 님` 버튼을 이용해 자동 로그인을 1회만 시도합니다.
5. 자동 로그인 시도 후에도 미복구면 임시 탭은 닫고, 현재 세션에 아래 형식으로 짧게 알리고 종료합니다.
```text
[로그인 해제 알림]
- 스마트스토어/톡톡 로그인 해제 상태
- 자동 로그인 1회 시도 후에도 미복구
- 조치 필요: 자동화 Chrome에서 로그인 확인
```
6. 같은 로그인 해제 상태를 매 실행마다 반복 길게 설명하지 말고, 실제 장애 알림만 짧게 유지합니다.

[3] 톡톡 목록 스냅샷 수집
1. 전용 자동화 Chrome(9223)에 연결합니다.
2. `talkUrl` 탭을 열거나 기존 탭을 재사용합니다.
2-1. 이 단계에서도 창을 앞으로 가져오지 말고 백그라운드 탭 기준으로 처리합니다.
3. 톡톡 목록에서 최근 대화 기준 상위 40개 이내를 수집합니다.
4. 각 항목마다 아래 값만 최소 수집합니다.
   - 대화 식별값(가능하면 popupPath / href / candidateId)
   - 고객 표시명
   - 마지막 미리보기 텍스트
   - 시간 라벨
   - 안읽음 개수
5. 결과를 `./runtime-data/talktalk-watch-state.latest.json` 형식으로 정리합니다.

[4] 변화 감지
1. 이전 상태 파일 `./runtime-data/talktalk-watch-state.json` 이 있으면 읽습니다.
2. 아래 기준 중 하나라도 바뀌면 변화로 봅니다.
   - 새 대화 식별값 등장
   - 같은 대화의 마지막 미리보기 텍스트 변경
   - 같은 대화의 시간 라벨 변경
   - 같은 대화의 안읽음 개수 증가
3. 변화가 없으면 `talktalk-watch-state.latest.json` 내용을 `talktalk-watch-state.json` 으로 반영하고 `NO_REPLY` 로 종료합니다.

[5] 변화 보고
1. 변화가 있으면 상위 10건까지만 요약합니다.
2. 배송문의로 보이는 건은 가능한 한 모두 개별로 적고, 초안 작성/승인/실제 발행 대상으로 이어질 수 있게 분명히 표시합니다.
3. 배송문의 외 항목은 개별 문장 나열 대신 문의유형별 건수만 짧게 적습니다.
4. 보고 형식은 아래처럼 짧게 유지합니다.
```text
[톡톡 watcher 변화 감지]
- 감지시각: YYYY-MM-DD HH:MM
- 변화 건수: N건
- 생성된 승인카드 수: N건
- 배송문의:

  - 고객명 | [신규/안읽음+2/문장변경] | 안읽음 3 | 시간 오후 11:18 | "문의드린 배송 언제 올까요"
  - 고객명 | [안읽음+1] | 안읽음 1 | 시간 오후 11:20 | "네 확인 부탁드립니다"

  [진행]
  - 위 배송문의 건은 초안 작성 대상입니다.
  - 각 건은 초안을 포함한 승인 카드로 올립니다.
  - 대표님 승인 후 실제 톡톡 발행까지 이어갑니다.

- 기타 문의 요약:
  - 일반문의 2건
  - 옵션/재고 문의 1건
- 메모: 목록 변화만 감지했고, 후속 배송문의 처리는 별도 단계에서 진행합니다.
```
5. 보고 후 `talktalk-watch-state.latest.json` 내용을 `talktalk-watch-state.json` 으로 반영합니다.

[6] 응답 규칙
1. 변화가 없으면 `NO_REPLY`
2. 변화가 있으면 위 형식대로만 짧게 보고
3. 실제 장애만 짧게 보고
