# WORKFLOWS.md — 운영 자동화 및 업무 흐름

## 목적
또몽이네 스토어의 반복 업무를 어떤 흐름으로 처리하는지 정리한다.
이 문서는 "무슨 업무가 어떤 도구로, 어떤 순서로, 어디서 막히는지"를 빠르게 파악하기 위한 운영 문서다.

## 1. CS 응대 흐름

### 대상 채널
- 네이버 톡톡
- 각 마켓 주문문의
- 상품 Q&A
- Discord 보고 채널 (내부 보고)

### 기본 흐름
0. 웹 검색/웹페이지 확인/로그인 사이트 조작이 필요한 업무는 일반 웹검색보다 Chrome Dev 기반 browser 도구를 우선 사용한다.
1. 고객 문의 확인
2. 문의 유형 분류
   - 배송 문의
   - 옵션/사이즈 문의
   - 교환/환불 문의
   - 통관 문의
   - 상품 정보 문의
3. 확인 가능한 정보 조회
4. 초안 작성
5. 민감도에 따라 즉시 답변 또는 대표 확인
6. 필요 시 이슈를 memory 또는 보고에 기록

### 사용 도구
- 크롬 확장 프로그램: CS 자동 답변 버튼 주입
- Quickstar Tracker 크롬 플러그인: 운송장/수취인/연락처/해외트래킹/주문번호 기반 조회
- tracking.tipoasis.com: 배송 추적
- 유니패스: 통관 보조 확인

### 핵심 규칙
- 인사말은 "안녕하세요, 또몽이네 스토어입니다."
- 고객응대 첫 문장은 기본적으로 `[마켓명]에서 주문하신 "[상품명]" 관련하여 안내 말씀 드립니다.` 형식 사용
- `주문해 주셔서 감사합니다` 문구는 사용하지 않음
- 상품명 강조는 큰따옴표(`""`) 사용
- 최종 고객응대 답변은 코드블럭으로 정리
- 주문정보에 010 번호가 있으면 별도 코드블럭으로 번호 분리 출력
- 확인 안 된 배송 상태는 추정 답변 금지
- 환불/보상 금액은 대표 승인 전 확정 금지
- 취소 안내에 `직접` 표현 금지
- 스마트스토어/쿠팡 취소 안내 시 취소 링크 포함
- 사이즈/색상은 상품 상세 기준으로만 안내
- 통관 오류는 개인통관고유부호 / 우편번호 / 전화번호 불일치 유형을 구분해서 설명

### 자주 막히는 지점
- 송장 상태 업데이트 지연
- 통관 단계 정보 불일치
- 고객이 주문정보를 충분히 주지 않음
- 마켓별 문의 UI가 달라 응답 흐름이 분산됨
- 톡톡 문의는 채팅형, 주문고객문의와 상품 Q&A는 1문1답형이라 자동화 방식이 다름

### 톡톡 문의 자동화 운영 원칙
- 네이버 톡톡 문의는 챗봇 API 웹훅 기반으로 실시간 수집한다.
- 상품 Q&A / 주문 고객문의는 커머스API + 크론 폴링 구조를 유지한다.
- 톡톡 수신 데이터는 GitHub가 아니라 로컬 저장소에 카드형으로 저장한다.
- 초기 톡톡 자동화는 자동응답 없이 수집 / 저장 / 조회 중심으로 운영한다.
- 고객별 카드에는 userId, 최근 메시지, 최근 시각, 메시지 개수, 수신 방향, 메시지 타임라인을 저장한다.
- 주문 이력이 있는 톡톡 고객은 파트너센터 팝업의 구매이력 영역에서 주문번호, 구매날짜, 옵션, 수량, 송장 등 주문 정보를 추가 추출한다.
- 웹훅은 채팅방 정밀 매칭 기준이 아니라 `새 문의 감지/알림 트리거` 용도로만 활용한다.
- 실제 처리의 기준 화면은 `https://sell.smartstore.naver.com/#/talktalk/chat` 이며, 안읽은 메시지 / 새로운 상담 / 미처리 상담 목록을 우선순위로 처리한다.
- 주문정보 확인과 CS 초안 작성은 스마트스토어 CS 톡톡 페이지 안 상담창/팝업 기준으로 진행한다.
- 크롬 확장은 필수 구성요소가 아니며, 반복 작업이 명확해질 때만 최소 기능으로 재도입한다.
- 승인 UI는 별도 웹페이지보다 Discord 중심으로 운영한다.
- `webhook.tipoasis.com` 은 웹훅 수신/저장/조회 트리거만 맡고, 승인 생성/초안 생성/Discord 보고/브라우저 반영은 모두 로컬에서 처리한다.
- 장기 목표는 `웹훅 감지 → 로컬 초안 생성 → Discord 승인 → 로컬 브라우저 반영` 구조다.

### 톡톡 배송문의 실무 순서
1. 스마트스토어 CS 톡톡 화면에서 상담을 연다.
2. 문의가 배송문의인지 먼저 분류한다.
3. 배송문의라면 퀵스타 탭 존재 여부와 로그인 상태를 먼저 확인한다.
   - 탭이 없으면 `https://quickstar.co.kr/`를 연다.
   - 로그인 안 되어 있으면 로그인 버튼을 눌러 로그인 완료 후 진행한다.
4. 필요 시 주문정보 팝업창을 열어 주문번호 / 상품주문번호 / 송장번호 / 구매이력을 확인한다.
5. 주문정보 파싱이 끝나면 팝업창은 바로 닫는다.
6. 톡톡 문의창으로 돌아와 Quickstar Tracker 조건 검색에 송장번호를 입력한다.
7. 조건 검색 결과가 채워질 때까지 기다린다.
8. 조건 검색 결과의 `현재상태 / 통관예상 / 배송예상 / 조회링크`를 기준으로 크롬 확장에서 고객 안내 초안을 생성한다.
9. 최근 대화 3~5개와 함께 **로컬에서** Discord 승인 카드 초안을 생성한다.
10. 대표 승인 후에만 실제 톡톡 반영을 진행한다.
11. 조건 검색 값이 비어 있으면 퀵스타 로그인 상태부터 다시 확인하고, 그래도 비면 tracking.tipoasis.com / 유니패스 / 국내택배를 보조 확인한다.

### Discord 승인 카드 운영 방식
1. 새 처리건이 감지되면 Discord 채널에 승인 대기 메시지를 올린다.
2. 승인 대기 메시지에는 아래 정보를 포함한다.
   - 문의 번호
   - 채널(톡톡/주문문의/상품문의)
   - 문의 유형
   - 고객명
   - 상품명
   - 주문번호 / 상품주문번호
   - 송장번호
   - 배송 상태 / 통관예상 / 배송예상
   - 최근 대화 3~5개
   - 생성 초안
3. 대표는 우선 텍스트 명령으로 승인한다.
   - `1번 승인`
   - `1번 보류`
   - `1번 수정: 마지막 문단만 짧게`
4. 이후 필요 시 Discord 버튼형 UI(`승인`, `보류`, `수정요청`)로 확장한다.
5. 승인된 건만 로컬 브라우저 자동화가 실제 톡톡/마켓 화면에 반영한다.

### Discord 버튼 클릭 후 한 번에 이어지는 로컬 처리 흐름
1. CS 승인 채널에서 버튼 클릭 이벤트가 들어오면 버튼 라벨/본문에 포함된 `승인코드`를 먼저 추출한다.
   - 예: `dd6075`
2. 액션을 분류한다.
   - `승인` → `approve`
   - `보류` → `hold`
   - `수정요청` → `revise`
3. 아래 명령으로 로컬 approval 파일 상태를 먼저 반영한다.
   - `node ./scripts/local-approval-action.mjs <승인코드> <approve|hold|revise> 대표님`
4. 명령 결과에서 아래 정보를 바로 받는다.
   - `approvalId`
   - `status`
   - `customerName`
   - `productName`
   - `draft`
   - `nextStep`
5. `status=approved` 이고 `nextStep=send_to_talktalk` 이면, 같은 흐름 안에서 바로 톡톡 파트너센터를 열어 해당 고객 상담을 찾고 초안을 입력/전송한다.
6. `hold` 또는 `revise` 이면 실제 톡톡 반영은 하지 않고 거기서 멈춘다.

## 2. 배송 조회 / 통관 확인 흐름

### 기본 흐름
1. 송장번호 또는 주문 정보 확보
2. Quickstar Tracker 또는 tracking.tipoasis.com 우선 조회
3. 필요 시 유니패스 보조 조회
4. 필요 시 CJ대한통운 등 국내 배송 상태 확인
5. 고객 안내 또는 내부 보고
6. 통관 보류/지연이면 즉시 대표 보고

### 사용 도구
- Quickstar Tracker 크롬 플러그인
- tracking.tipoasis.com
- 유니패스
- CJ대한통운 조회

### Quickstar Tracker 활용 흐름
1. 입력값 입력
2. 플러그인이 유형 자동 감지
   - 12자리 숫자 → 국내 운송장
   - GR 시작 → 그룹번호
   - 010- → 연락처
   - YT / SF / JD / 77 등 → 해외트래킹
   - 16자리 이상 숫자 → 해외주문번호
   - 한글 2~4자 → 수취인
3. 퀵스타 / 7customs / CJ 병렬 조회
4. 통관 상태, 반입 예정, 국내 배송 상태 조합
5. 고객 안내 초안 생성

### 핵심 규칙
- 배송 추적 링크 형식: `https://tracking.tipoasis.com/?t={송장번호}`
- 통관/세관 관련 내용은 법적 확정 표현 금지
- 확인되지 않은 구간은 단정 표현 금지
- 해외창고 / 통관 / 국내배송 단계를 혼동하지 않는다.
- 고객에게 내부 물류 상태를 그대로 노출하지 말고 이해 가능한 단계로 정리한다.

## 3. 상품 Q&A 반자동 승인 루프

### 대상 페이지
- 네이버 스마트스토어 상품 Q&A
- URL: `https://sell.smartstore.naver.com/#/comment/`

### 기본 흐름
1. 상품 Q&A 페이지 진입
2. 브라우저/MCP 연결 상태를 먼저 확인한다.
3. 반드시 `./scripts/browser-mcp/browser-ensure-ready.sh`를 먼저 실행해 `ready` 여부를 확인한다.
4. 연결이 살아 있으면 기존 세션을 그대로 사용한다.
4-1. 스마트스토어 관련 탭이 이미 열려 있으면 그 탭을 우선 사용하고, 없을 때만 새 탭을 연다.
5. 연결이 끊겼거나 실제 제어가 안 될 때만 attach/복구를 시도한다.
6. Google Chrome이 실행되어 있지 않으면 `./scripts/browser-mcp/browser-ensure-ready.sh`가 먼저 Chrome을 실행한 뒤 준비 상태를 다시 확인한다.
7. Chrome은 실행 중이지만 MCP attach 또는 권한 허용이 안 된 경우에는 `./scripts/browser-mcp/browser-ensure-ready.sh`로 attach 재시도 + 허용 팝업 자동 처리 + 재확인을 먼저 시도한다.
8. 그래도 실패하면 대표에게 `Chrome MCP attach 실패`와 실패 원인을 짧게 보고하고 종료한다.
9. attach 복구가 성공한 경우에만 이후 상품 Q&A 점검 단계로 넘어간다.
10. attach 복구가 실패하면 실패 원인을 보고하고 종료한다.
   - 점검 전 상태 확인 스크립트: `./scripts/browser-mcp/chrome-mcp-preflight.sh`
   - 브라우저 준비 보장 스크립트: `./scripts/browser-mcp/browser-ensure-ready.sh`
   - 팝업 감지/승인 모니터 스크립트: `./scripts/browser-mcp/chrome-dev-approve-monitor.sh`
   - 승인 후 attach 재시도 스크립트: `./scripts/browser-mcp/chrome-mcp-attach-approved.sh`
   - 승인 후 Q&A 점검 재개 스크립트: `./scripts/browser-mcp/qna-resume-after-approval.sh`
10-1. 상품 Q&A 외의 다른 browser 도구 작업도 동일하게 `./scripts/browser-mcp/browser-ensure-ready.sh`를 공통 선행 단계로 사용한다.
10-2. 스마트스토어 접근 시 로그인되지 않은 상태로 보이면 로그인 복구를 먼저 시도한다.
   - 구형 화면에서 `<span>로그인하기</span>` 버튼이 보이면 그 버튼을 클릭해 로그인 페이지로 이동한다.
   - 변경된 로그인 화면에서 `<span>로그인</span>` 버튼이 보이고 아이디/비밀번호가 이미 채워져 있으면 해당 버튼을 클릭해 로그인 완료를 시도한다.
   - 로그인 완료 후 스마트스토어센터 메인/문의 화면 진입이 확인되면 이후 업무를 계속 진행한다.
   - 로그인 버튼 클릭 후에도 로그인되지 않으면 그때만 대표님께 로그인 상태를 짧게 보고한다.
12. 답변 필터를 `미답변`으로 설정한다.
13. `검색` 버튼 클릭
14. 미답변 목록 추출
15. 상품명 + 질문내용 기반으로 초안 작성
16. Discord 대화방에 승인 요청 보고서 작성
17. 대표가 항목별 승인 / 수정 / 보류 지시
18. 승인된 항목만 브라우저에 반영
19. 답글 등록 후 `저장되었습니다` 팝업의 `확인` 버튼 클릭
20. 등록 완료 후 다시 `미답변` 검색을 실행한다.
21. 재검색된 최신 화면을 캡처한다.
22. Discord 대화방에 완료 보고를 작성하고, 최신 화면 캡처 파일을 **이 채널에 직접 첨부**한다.
23. 다음 루프를 진행한다.

### 승인 요청 보고 방식
- 디스코드에서 항목 번호 기준으로 보고
- 예시:
  - `1번 승인`
  - `1,2번 승인`
  - `2번 수정`
  - `전체 보류`
- 보고는 표 또는 카드형 목록으로, 상품명 / 질문 / 초안을 보기 쉽게 정리한다.

### 승인 후 완료 보고 방식
- 승인된 항목만 등록한다.
- 등록 완료 후 `미답변` 재검색을 실행한다.
- 최신 결과 화면을 캡처해 Discord에 첨부한다.
- 완료 보고에는 최소한 아래를 포함한다.
  - 처리한 승인 번호
  - 등록 완료 여부
  - 재검색 완료 여부
  - 최신 화면 캡처 파일 첨부 여부
- 캡처는 파일 경로만 남기지 말고, **현재 Discord 채널에 실제 첨부 전송**해야 완료로 본다.

### 브라우저 입력 규칙
- `답글` 버튼을 먼저 클릭해서 입력창을 연다.
- textarea는 단순 fill로는 Angular 모델 반영이 안 될 수 있다.
- 값 주입 후 `input`, `change`, `blur` 이벤트까지 발생시켜야 한다.
- 글자 수가 `0/1000`에서 변경되었는지 확인 후 등록한다.

### 자동 실행 정책
- 상품 Q&A 미답변 조회는 **1시간에 1번** 점검 대상이다.
- 점검 시간은 **한국시간 09:00~22:00** 범위 안에서만 적용한다.
- 점검 시 자동 등록하지 않고, **항상 대표 승인 요청부터** 진행한다.
- 승인 후 후속처리에서는 등록 완료 뒤 재검색 + 화면 캡처 + Discord 첨부 완료 보고까지 수행한다.
- 실제 1시간 주기 실행은 별도 스케줄러/하트비트 설정이 필요하며, 시스템 변경 전에 대표 확인 후 적용한다.

## 9. Discord 운영 흐름

### 기본 흐름
1. 대표 지시 수신
2. 요청 유형 파악
   - 보고 요청
   - 조사/검토 요청
   - 문서 수정 요청
   - git / 백업 요청
   - 웹 브라우징 요청
3. 실행
4. 요약 보고
5. 의미 있는 문서 변경이면 로컬 git 커밋
6. 의미 있는 문서 변경이면 push까지 진행

### 브라우징 요청 처리 원칙
1. 브라우저/MCP 연결 상태 먼저 점검
2. 연결이 살아 있으면 바로 진행
3. 연결이 끊겨 있으면 대표에게 먼저 알림
4. 필요 시에만 Chrome 실행 또는 attach 복구를 진행
5. 불필요한 재attach는 하지 않는다.
6. attach 복구 후 작업 진행

### 핵심 규칙
- 그룹 채널에서는 민감정보 상세 노출 최소화
- 짧고 직접적으로 보고
- 실행 전후 상태 차이를 명확히 설명

## 10. Git / 백업 흐름

### 기본 흐름
1. workspace 문서/설정 변경
2. `git status` 확인
3. 의미 있는 변경 묶음으로 `git commit`

- `memory/`, `MEMORY.md`, `.openclaw/`, `.env*`는 기본 제외
