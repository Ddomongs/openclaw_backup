# WORKFLOWS.md — 운영 자동화 및 업무 흐름

## 목적
또몽이네 스토어의 반복 업무를 어떤 흐름으로 처리하는지 정리한다.
이 문서는 "무슨 업무가 어떤 도구로, 어떤 순서로, 어디서 막히는지"를 빠르게 파악하기 위한 운영 문서다.

---

## 1. CS 응대 흐름

### 대상 채널
- 네이버 톡톡
- 각 마켓 주문문의
- 상품 Q&A
- Discord 보고 채널 (내부 보고)

### 기본 흐름
1. 고객 문의 확인
2. 문의 유형 분류
   - 배송 문의
   - 옵션/사이즈 문의
   - 교환/환불 문의
   - 통관 문의
   - 상품 정보 문의
3. 확인 가능한 정보 조회
4. 초안 작성
5. 민감도에 따라 즉시 답변 또는 대표 확인
6. 필요 시 이슈를 memory 또는 보고에 기록

### 사용 도구
- 크롬 확장 프로그램: CS 자동 답변 버튼 주입
- tracking.tipoasis.com: 배송 추적
- 유니패스: 통관 보조 확인

### 핵심 규칙
- 인사말은 "안녕하세요, 또몽이네 스토어입니다."
- 확인 안 된 배송 상태는 추정 답변 금지
- 환불/보상 금액은 대표 승인 전 확정 금지
- 사이즈/색상은 상품 상세 기준으로만 안내

### 자주 막히는 지점
- 송장 상태 업데이트 지연
- 통관 단계 정보 불일치
- 고객이 주문정보를 충분히 주지 않음
- 마켓별 문의 UI가 달라 응답 흐름이 분산됨

---

## 2. 배송 조회 / 통관 확인 흐름

### 기본 흐름
1. 송장번호 또는 주문 정보 확보
2. tracking.tipoasis.com 우선 조회
3. 필요 시 유니패스 보조 조회
4. 고객 안내 또는 내부 보고
5. 통관 보류/지연이면 즉시 대표 보고

### 사용 도구
- tracking.tipoasis.com
- 유니패스

### 핵심 규칙
- 배송 추적 링크 형식: `https://tracking.tipoasis.com/?t={송장번호}`
- 통관/세관 관련 내용은 법적 확정 표현 금지
- 확인되지 않은 구간은 단정 표현 금지

## 3. 상품 등록 흐름

### 대상 플랫폼
- 퍼센티(Percenty)
- 네이버 스마트스토어
- 쿠팡
- 지마켓
- 11번가

### 기본 흐름
1. 소싱 상품 확인
2. 상품명/옵션/이미지/상세정보 정리
3. 가격 산출
4. 퍼센티 기준으로 등록 데이터 구성
5. 배치 업로드
6. 업로드 결과 점검
7. 실패 건 재처리 또는 수정

### 사용 도구
- 퍼센티(Percenty)
- percenty-auto-upload.js

### 핵심 규칙
- 가격 산출 포함 항목: 원가 + 국제배송비 + 관세/부가세 + 국내배송비 + 마진
- 환율은 당일 기준 + 3~5% 여유 반영
- 배치 업로드는 50개 단위, 20분 타임아웃 전제

### 자주 막히는 지점
- 옵션 정리 불완전
- 이미지/상세페이지 포맷 불일치
- 퍼센티 업로드 타임아웃
- 마켓별 필수값 누락

---

## 4. 가격 산출 흐름

### 기본 흐름
1. 상품 원가(CNY) 확인
2. 국제배송비 반영
3. 관세/부가세 여부 판단
4. 국내배송비 반영
5. 목표 마진 반영
6. 최종 판매가 산정

### 확인 포인트
- HS 코드 분류 필요 여부
- FTA/RCEP 적용 가능성
- 인증 필요 품목 여부

### 핵심 규칙
- 가격은 감으로 정하지 않는다.
- 수치 계산 시 누락 항목이 없도록 한다.
- 민감 품목은 통관 리스크를 먼저 본다.

---

## 5. 파트타임 정산 흐름

### 기본 흐름
1. 기간 확인
2. 상품 등록 처리 건수 확인
3. 건당 단가 적용
4. 총 건수 / 총 금액 / 기간 기준으로 보고

### 핵심 규칙
- 정산 보고에는 반드시 기간 명시
- 총 건수와 총 금액을 함께 제시
- 계산 기준이 바뀌면 대표 확인 후 반영

---

## 6. 카드 명세 분류 흐름

### 기본 흐름
1. 카드 결제 내역 확인
2. 외화 / 원화 여부 분류
3. 외화는 사업용 우선 분류
4. 원화는 건별 확인
5. 애매한 내역은 보류 후 대표 확인

### 핵심 규칙
- 추정 분류 금지
- 애매한 건은 메모 남기고 보류

## 7. 정기 점검 / 브리핑 흐름

### 매 하트비트 확인
- 미답변 CS 문의 여부
- 통관 보류 건 여부
- 긴급 클레임 발생 여부

### 아침 브리핑 (09:00 KST)
- 어제 처리/미처리 CS 건수
- 배송 중 주문 현황
- 통관 상태 이상 유무
- 오늘 우선순위 업무

### 저녁 마감 (21:00 KST)
- 오늘 처리 완료 건수
- 미처리 건 목록과 사유
- 특이사항 / 이슈를 memory/YYYY-MM-DD.md에 기록

### 핵심 규칙
- 정기 보고는 요약 먼저
- 긴급 이슈는 즉시 별도 보고
- 반복 오류는 중복 보고하지 말고 최초 보고 여부 확인

## 8. Discord 운영 흐름

### 기본 흐름
1. 대표 지시 수신
2. 요청 유형 파악
   - 보고 요청
   - 조사/검토 요청
   - 문서 수정 요청
   - git / 백업 요청
3. 실행
4. 요약 보고
5. 의미 있는 문서 변경이면 로컬 git 커밋
6. push는 대표 요청 시만 진행

### 핵심 규칙
- 그룹 채널에서는 민감정보 상세 노출 최소화
- 짧고 직접적으로 보고
- 실행 전후 상태 차이를 명확히 설명

---

## 9. Git / 백업 흐름

### 기본 흐름
1. workspace 문서/설정 변경
2. `git status` 확인
3. 의미 있는 변경 묶음으로 `git commit`
4. 대표 요청 시 `git push`

### 핵심 규칙
- 운영 방식은 반자동
- 로컬 커밋은 기본, 원격 push는 요청 시
- `memory/`, `MEMORY.md`, `.openclaw/`, `.env*`는 기본 제외

## 10. 자동화 개선 후보

### 우선순위 높음
- CS 자동응답 흐름 표준화
- 퍼센티 업로드 실패 패턴 정리
- 상품 등록용 체크리스트 문서화
- 브리핑용 집계 포맷 고정

### 추후 검토
- 외부 응대 전용 분리 구조
- 자동화별 실패 로그 문서화
- 마켓별 API 연동 흐름 세분화

---

## 운영 메모
- 자동화는 가능하면 API → 스크립트 → GUI 순으로 우선순위를 둔다.
- 같은 작업이 여러 경로에서 중복 실행되지 않게 한다.
- 중요한 운영 판단은 세션에만 두지 말고 문서에 남긴다.
